## 1. 배경 및 문제 정의

### 1.1 배경

금융감독원(FSS)은 금융회사·임직원에 대한 **제재공시**와 검사 결과 **경영유의·개선사항**을 홈페이지에 부정기적으로 게시한다. 타 금융회사의 실제 제재사례는 IBK기업은행이 **동일한 통제 미비점을 선제적으로 자가점검**할 수 있는 가장 구체적인 벤치마킹 자료다.

### 1.2 문제 정의 (AS-IS)

| # | 문제 | 영향 |
|---|---|---|
| P1 | FSS 공시가 **부정기적**이라 담당자가 매일 수동으로 확인해야 함 | 반복 수작업, 누락 위험 |
| P2 | 공시 목록에 과거 건이 누적 노출되어 **신규 건 식별이 어려움** | 중복 검토 또는 신규 건 누락 |
| P3 | 제재 원문(HTML·PDF)이 길고 법률적 표현 위주 | 건별 분석에 시간 소요, IBK 업무 연관성 판단이 담당자 역량에 의존 |
| P4 | 은행 제재와 환전영업소 제재가 같은 목록에 섞여 노출 | 중요도 선별 없이 전건 검토하는 비효율 |
| P5 | 검토 이력이 개인 메모 수준으로 산재 | 감사 추적(audit trail) 부재 |

### 1.3 기회

자매 프로젝트 **FSC 입법예고 브리핑(Daily-Morning-brief)** 에서 완전 클라우드 멀티에이전트 파이프라인(트리거→수집→LLM 분석→보고서→검증→아카이브→알림)의 골격이 검증 완료되어 있었다. 데이터 수집 계층과 분석 관점만 FSS 제재 도메인으로 신규 작성하면 낮은 비용으로 동일 품질의 자동화가 가능했다.

## 2. 추진 방향 (Direction)

> **"금감원이 제재·경영유의 건을 게시하면, 신규분만 자동 수집해 Claude LLM이 IBK 업무 연관성(벤치마킹 자가점검 관점)을 분석하고, 매일 08:00·16:00 Telegram 알림과 DOCX 보고서로 전달한다. 전 과정은 로컬 PC 없이 100% 클라우드에서 실행한다."**

### 2.1 핵심 방향 4가지

1. **사후(事後) 벤치마킹 관점** — FSC 프로젝트(예방: 법령 변경 대응)와 달리, 본 프로젝트는 *실제 발생한 제재사례*를 기반으로 "IBK에도 같은 통제 미비점이 있는가"를 점검하는 **사후 자가점검** 도구다. 분석 프롬프트·중요도 로직 전체가 이 관점으로 설계된다.
2. **완전 클라우드** — Cloudflare Workers Cron + GitHub Actions 단일 Job으로 로컬 PC·상시 서버 없이 동작한다. 상태(중복방지 원장)는 git 저장소 자체에 보존한다.
3. **AI(LLM)는 분석에만, 판단 어조는 제안형으로** — 제재는 법적으로 민감하므로 LLM 분석은 단정을 금지하고 "점검 제안"형 해요체로만 표현한다(tone-guide 주입 + validator 검증의 이중 안전장치).
4. **감사 추적 가능성(Auditability)** — 수집 원본(HTML·PDF), 분석 결과, 실행 메타를 매 실행 git 커밋으로 남겨 내부통제 업무 특성에 맞는 증빙 체계를 갖춘다.

### 2.2 하지 않을 것 (Non-Goals)

- 제재사례에 대한 **법적 판단·단정적 결론 제시** (점검 "제안"까지만)
- FSC 입법예고 브리핑과의 **채널·코드 통합** (별개 repo·별개 Telegram 봇 유지)
- 실시간(게시 즉시) 알림 — 일 2회(08:00·16:00 KST) 배치로 충분
- 사내망 시스템 연동 (현 단계는 Telegram + DOCX 전달까지)

## 8. 기대 효과

- **수작업 제거**: 매일 FSS 홈페이지 2개 게시판 수동 확인 → 0분 (알림 수신으로 대체)
- **누락·중복 차단**: ledger 기반 신규 판별로 신규 건 누락과 재알림을 구조적으로 방지
- **점검 품질 표준화**: 건별로 "무슨 일 → IBK 발생 가능성 → 점검 제안 → 담당 부서"가 일관된 틀로 제공
- **선제적 리스크 대응**: 타행 제재사례를 IBK 자가점검 트리거로 전환 (사후 → 선제)
- **감사 대응력**: 실행·수집·분석 전 이력이 git으로 증빙 가능

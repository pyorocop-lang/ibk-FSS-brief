## Part 1. 비개발자용 (경영진·현업 부서)

### Q1. 이 시스템은 한 줄로 무엇을 하나요?
금융감독원이 제재·경영유의 건을 게시하면, 신규 건만 골라 AI가 "IBK에도 같은 일이 생길 수 있는가"를 분석해 매일 아침 8시와 오후 4시 Telegram 알림과 워드(DOCX) 보고서로 보내줍니다. 담당자가 금감원 홈페이지를 매일 확인하던 수작업이 사라집니다.

### Q2. 기존 FSC 법령 브리핑과 무엇이 다른가요?
FSC 브리핑은 **예방**입니다 — 법이 바뀌기 전에 미리 대응하는 것. 이 시스템은 **사후**입니다 — 다른 금융회사가 실제로 받은 제재를 보고 "우리도 같은 허점이 있나" 자가점검하는 벤치마킹 도구입니다. 그래서 별도 저장소·별도 알림 봇으로 완전히 분리 운영합니다.

### Q3. AI가 하는 일과 사람이 하는 일은 어떻게 나뉘나요?
AI(Claude)는 **분석문 작성만** 합니다 — 제재 요약, IBK 발생 가능성, 점검 제안, 담당 부서 추천, 위험도. 수집·신규 판별·보고서 조립·검증·보존은 전부 일반 프로그램이 수행합니다. 그리고 **점검 실행 여부의 최종 판단은 항상 담당자**가 합니다. AI 분석은 참고자료이지 법적 판단 근거가 아닙니다.

### Q4. AI가 틀리면 어떻게 되나요?
세 겹의 안전장치가 있습니다. ① AI는 단정을 못 하도록 설계돼 있어 "~점검을 제안해요" 같은 제안형으로만 말합니다. ② 검증 프로그램(validator)이 톤·누락·구조를 매 실행 자동 점검합니다. ③ 제재 원문 PDF가 항상 함께 보관되므로 담당자가 원문 대조로 확인할 수 있습니다. 중요 건(IBK 직접·위험도 상)은 원문 확인 후 조치하는 것이 운영 원칙입니다.

### Q5. 왜 매일 오는데 어떤 날은 "신규 없음" 한 줄만 오나요?
금감원 제재 공시는 부정기적이라 신규가 없는 날이 많습니다. "✅ 신규 없음"은 **시스템이 정상 작동해서 확인까지 마쳤다**는 뜻입니다. 반대로 수집에 실패하면 "신규 없음"이 아니라 "❌ 수집 실패"라고 명확히 구분해서 알려줍니다 — 실패를 "없음"으로 오인시키지 않는 것이 설계 원칙입니다.

### Q6. 알림에서 어떤 건 빠지고 어떤 건 들어가나요?
제재받은 기관의 계층으로 거릅니다. IBK 직접(T0)·은행(T1)·인접금융(T2, 보험·증권·카드 등)은 전건 알림에 포함되고, 환전영업소·대부업체·보험대리점 같은 주변 업권(T3)은 알림에서 빼되 보고서에는 전건 수록합니다. IBK와 무관한 소규모 제재로 알림이 시끄러워지는 것을 막기 위해서입니다.

### Q7. 운영 비용은 얼마나 드나요?
사실상 무료 수준입니다. GitHub Actions·Cloudflare Workers는 무료 티어 내에서 돌고, AI는 가장 저렴한 경량 모델(Haiku)을 쓰며 **신규 건만** 분석하므로(하루 0~수 건) API 비용이 미미합니다. 별도 서버·PC가 없어 유지비도 없습니다.

### Q8. 담당자 PC가 꺼져 있어도 되나요?
네. 전 과정이 클라우드(Cloudflare + GitHub)에서 실행됩니다. 로컬 PC는 어떤 단계에도 관여하지 않습니다.

### Q9. 보고서는 어디서 받나요?
매일 알림은 Telegram으로 옵니다. DOCX 보고서와 제재 원문 PDF는 GitHub Actions의 Artifact(`fss-brief-날짜-am`(오전)·`fss-brief-날짜-pm`(오후))에서 다운로드하며 90일 보관됩니다.

### Q10. 감사(監査) 대응이 되나요?
네, 오히려 강점입니다. 매 실행의 수집 원본(HTML·PDF)·분석 결과·실행 기록이 git·Artifact에 자동 보존되어 "언제 무엇을 수집해 어떻게 보고했는지"가 전부 증빙됩니다. **신규가 없는 날(변동 없음)에도** 크롤 증거가 남습니다: `crawl_result.json`(크롤 시각·스캔 총건수·관측 창)과 그 안의 `scanAudit`(스캔한 목록 key 전체 + 페이지 본문 SHA-256 해시)가 **git에 영구 커밋**되므로, 원본 HTML(Artifact 90일)이 만료돼도 "그날 게시판에서 무엇을 스캔했는지"를 항구적으로 증적할 수 있습니다.

### Q11. 이 시스템이 담당 인력을 대체하나요?
아니요. 모니터링·요약·초안 작성이라는 반복 작업을 대체하고, 담당자는 **판단과 점검 실행**에 집중하게 하는 도구입니다. 점검 제안을 채택할지, 유관 부서에 전파할지는 사람이 결정합니다.

### Q12. 왜 하루 두 번(오전·오후) 오나요? 오후엔 무엇이 다른가요?
오전 8시(am)에는 **전날 오후(pm) 확인 이후 새로 게시된** 신규 제재·경영유의 건을 알립니다. 오후 4시(pm)에는 **오전(am) 확인 이후** 새로 올라온 건만 델타로 알리고, 새 건이 없으면 "변동 없음 · 기존 점검 유지" 한 줄로 마감합니다(중복방지 원장이 매 실행 갱신·커밋되므로 두 실행은 연속된 델타 창입니다 — am=전날 오후 이후, pm=오늘 오전 이후). FSS 제재는 부정기 발행이라 오후는 대개 조용한 마감입니다. FSC 법령 브리핑과 동일한 오전/오후 커버리지를 맞추기 위한 구성입니다.

### Q13. "오늘의 브리핑"인데 알림에 뜬 건의 날짜가 오늘이 아니던데요? "신규"는 무슨 기준인가요?
"오늘"은 **브리핑을 만든 날(우리가 새로 확인·수집한 날)** 을 뜻하며, 각 건에 적힌 **제재조치요구일**과는 다른 개념입니다.

금감원 제재 목록에는 **'게시일' 칸이 없습니다.** 목록은 제재조치요구일 순으로 정렬되기 때문에, 오늘 새로 올라온 건이라도 조치요구일이 지난 날짜면 목록 맨 위가 아니라 **중간에 꽂힙니다.** 그래서 날짜만으로는 신규를 가려낼 수 없습니다.

처음에는 '게시일 기준일(앵커)' 방식으로 신규를 가렸습니다. 그런데 목록의 정렬 기준(조치요구일)에 커트오프를 건 탓에, **늦게 게시된 과거 조치요구일 건이 조용히 버려졌습니다.** 실제로 유일한 IBK 계열 건(`아이비케이신용정보` — 2026-07-09 새로 게시, 조치요구일 06-25)을 놓쳤습니다. 이 미탐을 발견해 **2026-07-10 앵커를 폐지**했습니다.

지금은 **직전 실행이 그때 무엇을 봤는지**를 스캔 증적으로 남겨두고, 이번에 본 목록과 비교해 **그 사이 새로 나타난 건만** 신규로 봅니다(관측 창 차집합). 이미 본 건은 중복방지 원장이 걸러 다시 알리지 않습니다. 직전에 훑지 않은 깊이(창 밖)나 최초 실행분은 과거 누적분이므로 원장에만 등록하고 보고에서 제외하되, **조용히 버리지 않고 기록으로 남깁니다.**

신규 건이 목록 상단에 없어 헷갈릴 수 있으므로, 해당하는 건에는 알림·보고서 맨 끝에 **'목록에서 찾는 방법'**(조치요구일 · 최초 등장일 · 목록 행 번호)을 함께 안내합니다.

## Part 2. 개발자용

### Q1. 왜 GitHub schedule cron 대신 Cloudflare Workers Cron인가?
GitHub `schedule` cron은 최대 ~12시간 지연·누락이 실측됐습니다. 08:00·16:00 정시성이 요구사항이라 외부 Cloudflare Workers Cron(am `0 23 * * *` · pm `0 7 * * *` UTC)이 `workflow_dispatch`를 호출하는 구조로 대체했고, GitHub schedule은 백업으로도 남기지 않았습니다(이중 발화 방지).

### Q2. 상태(중복방지 원장)를 왜 DB가 아니라 git에 두나?
러너가 매 실행 휘발되는 완전 클라우드 구조라 **repo가 유일한 영속 저장소**입니다. `state/seen_ids.json`을 성공 시에만 커밋하는 방식은 ① 별도 인프라 0원 ② 상태 변경 이력이 곧 감사 로그 ③ 장애 시 git revert로 복구 가능이라는 이점이 있습니다. 일 수 건 규모라 동시성·용량 문제가 없습니다.

### Q3. dedup 키는 어떻게 설계했나?
소스별 실측 고유 식별자를 씁니다. 제재공시는 상세 URL 쿼리의 `examMgmtNo_emOpenSeq`, 경영유의는 상세페이지가 없어 첨부 PDF 파일명 선두 ID(예: `202600082_11`)입니다. 제목 해시가 아닌 원천 식별자라 제목 수정에도 안정적입니다.

### Q4. 수집 실패 격리(failure isolation)는 왜 이런 구조인가?
2026-06-27에 timeout 실패 런이 그날의 성공 기록 6건을 0건으로 덮어쓴 사고가 있었습니다(f93e0c4). 이후 계약을 바꿨습니다: **실패 시 크롤러는 `failure_meta.json`만 쓰고 `crawl_result.json`·ledger는 절대 건드리지 않는다.** 워크플로우 STEP6는 failure_meta 존재 여부로 커밋 대상을 갈라 성공 데이터와 실패 로그를 분리 커밋합니다. 덕분에 실패가 "신규 없음" 오보나 데이터 유실로 이어질 수 없습니다.

### Q5. 왜 Haiku인가? 더 큰 모델을 쓰지 않는 이유는?
작업이 "긴 원문 → 정형 11필드 요약·매핑"이라 경량 모델로 충분하고, 부족한 도메인 맥락은 프롬프트에 IBK 지식(`knowledge/` 5종)을 주입해 보완합니다. 비용·속도 대비 품질이 목표치를 만족했고, `ANALYST_MODEL` env로 언제든 상위 모델로 교체 가능한 구조입니다.

### Q6. MAX_TOKENS가 2048인 이유는?
1024로는 한국어 11필드 JSON이 잘려 미완결 JSON→파싱 실패→폴백 강등되는 사례가 실측됐습니다(2026-07-02 우리은행 건). Haiku라 토큰 비용이 무시 가능해 2048로 상향했습니다.

### Q7. LLM 병렬도가 3인 근거는?
직렬 병목을 피하되 Haiku Tier1 rate limit(50 RPM) 여유 내에서 동작하도록 `CONCURRENCY = 3`으로 제한했습니다. 분석 대상이 신규 건뿐이라(보통 0~수 건) 이 이상의 병렬화는 실익이 없습니다.

### Q8. analyst의 exit code 규약은?
`0`=정상, `1`=fallback(API 키 미설정·API 오류 → 키워드 템플릿으로 강등, 파이프라인 계속 + warning), `2`=치명(시스템 프롬프트 파일 부재 등 → Job 중단). "분석 품질 저하"와 "실행 불가"를 구분해 가용성을 우선한 설계입니다.

### Q9. STEP6 push에서 `git pull --rebase -X theirs`를 쓰는 이유는?
같은 날짜 재실행 시 `crawl_result.json`이 add/add 충돌하는데, 일반 rebase는 충돌 마커가 JSON에 박혀 완료 알림의 `JSON.parse`가 깨지는 것이 로컬 재현으로 확인됐습니다. `-X theirs`는 최신 런(현재 실행분)을 채택해 exit 0·유효 JSON을 보장합니다. rebase가 그래도 중단되면 `rebase --abort`로 깨끗한 상태로 복구 후 재시도(3회)합니다.

### Q10. 최초 실행 때 과거 공시가 전부 알림으로 쏟아지지 않나?
시드 모드로 방지합니다. ledger가 비어 있으면 crawler는 목록·ledger만 채우고 분석 대상(`graded[]`)은 비웁니다. 초기엔 이 격리가 없어 과거 누적건이 전부 '신규'로 범람한 버그가 있었고 2026-07-01에 수정됐습니다.

### Q11. FSS가 해외 IP를 막으면 어떻게 하나? (GitHub 러너는 미국)
사전 진단 워크플로우(`diag-fss-access.yml`)로 미국 러너 4종에서 접근 PASS를 확인했습니다(결정 C). 그래서 KR 프록시·중계 계층 없이 직결 스크래핑합니다. 만약 향후 차단되면 수집 3회 재시도 후 "❌ 수집 실패" 알림으로 즉시 드러나므로, 그때 프록시 계층을 추가하면 됩니다.

### Q12. 왜 OPEN API가 아니라 스크래핑인가?
FSS OPEN API에는 제재/경영유의 엔드포인트가 존재하지 않습니다(결정 D). 대신 파서는 목록 HTML의 앵커 href에서 상세 경로를 추출하는 방식이라(URL 패턴 추정·하드코딩 없음) 소소한 개편에는 상대적으로 견고합니다.

### Q13. tgMsg·보고서 품질은 어떻게 회귀 방지하나?
이중 장치입니다. ① 생성 측: tone-guide(해요체 8원칙)를 analyst 시스템 프롬프트에 주입 — 하드코딩 후처리가 아니라 생성 단계에서 톤을 잡습니다. ② 검증 측: validator가 A(톤 8원칙: 핵심 선행·문장 길이·복붙 금지·동사 종결·해요체 등)·B(절삭)·C(tgMsg 구조)·D(DOCX 구조 — 기대 섹션을 데이터로 계산해 실제 출력과 대조)를 매 실행 검사합니다.

### Q14. 파이프라인을 로컬에서 재현하려면?
`.env.example` 기준으로 env를 넣고 STEP 순서대로 실행하면 됩니다: `node fss_crawler.js --date YYYYMMDD` → `analyst.js` → `briefV2.js` → `validator.js` (동일 `--date`). 산출물은 `reports/{DATE}/{SLOT}/`에 생성되며 슬롯은 실행 시각(KST)으로 자동 결정됩니다(`runslot.js`).

### Q15. 자매 프로젝트(Daily-Morning-brief) 코드와 얼마나 겹치나?
briefV2·validator·archivist·notify_telegram·runslot·cloud-trigger 골격은 차용(+경미 조정)이고, **fss_crawler.js(수집·dedup·Tier)와 analyst.js 프롬프트(사후 벤치마킹 관점)는 FSS 전용 신규**입니다. 자매 repo는 읽기 전용 참조 원칙 — 직접 수정·push 금지입니다.

### Q16. 신규 판별을 어떻게 하나? 날짜로 자르면 되는 것 아닌가?
날짜로 자르면 **안 됩니다.** FSS 목록엔 게시일 컬럼이 없고, 3번째 컬럼인 `제재조치요구일`의 **내림차순**으로 정렬됩니다. 오늘 새로 게시된 건도 조치요구일이 과거면 목록 **중간에 삽입**되므로, 정렬 컬럼에 커트오프를 걸면 늦게 게시된 과거 조치요구일 건이 **침묵 폐기**됩니다(실제로 게시일 앵커 `REPORT_SINCE`가 `아이비케이신용정보` 건을 놓쳐 2026-07-10 폐지됐고, env로 설정해도 지금은 무시+경고입니다).

현행은 **직전 실행 관측 창 차집합(scan-window diff)** 입니다.
```
buildScanWindow(직전 crawl_result.scanAudit)
    → 소스별 { 본 key 집합, 훑은 깊이(depth) }

classifyRow(key, page, ledgerMap, win, seed) → known | new | backfill | skip
    레저/직전 창에 있음               → known    (재알림 차단)
    seed(최초 실행)                   → backfill (목록 전체가 과거 누적분)
    page ≤ win.depth 이고 레저에 없음 → new      (조치요구일이 과거여도 신규)
    page >  win.depth                 → backfill (창 밖 — 판정 근거 없음)
    직전 창 유실 시 depth = WINDOW_FALLBACK_DEPTH 가정
```
`backfill`은 `state/seen_ids.json`에만 등록하고 보고에서 제외하되 **`crawl_result.backfilled[]`에 명시 기록**합니다 — 판정 근거 없는 건을 **조용히 버리지 않는다**는 원칙입니다. 산출물은 `crawl_result.{newItems[], backfilled[], scanWindow{}, completeness{}, scanAudit[]}` + `newItemBasis: "scan-window-diff"`.

관측 창 깊이는 `--pages` **기본 5**(`FSS_MAX_PAGES`)이고, `scanWindow.floorLookbackDays`가 **45일** 미만이면 "창이 얕다"고 경고합니다(창이 얕으면 늦게 게시된 과거 조치요구일 건이 창 밖에 떨어져 영구 미탐). 수집 파이프라인은 두 소스 공통 `collectSource`로 `scanSource` → `classifySnapshot` → `reconcile`(총건수 체크섬 `listTotal − prevListTotal == 신규 − 삭제`) → `buildEntry` 순이며, 체크섬 불일치나 커버리지 미확립 시 전 페이지 심화 스캔으로 승격합니다.

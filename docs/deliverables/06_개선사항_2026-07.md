## 건 1. 오전·오후 하루 2회 발화 + 오후 델타 알림

### 📋 업무 문서 (경영진)
- **배경/문제:** 기존 08:00 1회 발화만으로는, 오전 이후(전날 저녁·당일 오전) 금감원이 새로 게시한 제재를 다음 날까지 놓쳤다.
- **개선 효과:** 자매 프로젝트(FSC 법령 브리핑)와 동일하게 **08:00·16:00 하루 2회** 발화. 오후 실행은 **오전 이후 새로 게시된 건만** 알린다. 제재 대응 착수 시점이 최대 하루 앞당겨진다.
- **부담 없음:** 오후에 신규가 없으면 조용한 '변동 없음' 마감 한 줄이라 알림 피로가 없다(FSS 제재는 부정기 발행이라 대개 조용한 마감).
- **리스크/대응:** 중복 알림 우려 → 중복방지 원장으로 원천 차단(같은 건 두 번 안 감).

### 🔧 기술 문서 (실무자)
- **트리거:** Cloudflare Workers Cron에 `0 7 * * *`(=16:00 KST) 추가 → `["0 23 * * *", "0 7 * * *"]`. 대시보드에 두 cron 등록(실발화 확인). Worker는 두 cron 모두 동일하게 `workflow_dispatch` 호출.
- **슬롯:** 러너가 KST 시각으로 판별(`runslot.js`, `<12=am` / `≥12=pm`). 산출물은 `reports/{DATE}/{am|pm}/`로 분리 보존(공존·비파괴).
- **오후 델타 게이트:** `daily-brief.yml` 완료알림 스텝 — pm이고 오전본이 있으면 `notify_telegram.js --delta-since reports/{DATE}/am/crawl_result.json` 호출. 오전 이후 신규만 알림, 0건이면 '변동 없음' 마감(시작→완료 짝 보장). am이거나 오전본 없으면 평소대로 전체 전송(놓침 방지).
- **키 정정:** 델타 비교키를 `noticeId`(FSS 미존재)→`key`(examMgmtNo_emOpenSeq / 파일ID)로 수정, FSC 호환 fallback.
- **검증:** 3경로(신규有 / 0 / 오전본없음) 실코드 리허설 통과. 오늘 am·pm 자동실행 커밋으로 라이브 실증.

## 건 4. 감사 증적 강화 — `crawl_result.scanAudit`

### 📋 업무 문서 (경영진)
- **배경/문제:** 신규가 없는 날의 원본 목록 HTML은 GitHub Artifact에 **90일만** 보관 → 감사자가 90일 초과 시점에 "그날 게시판을 실제로 확인했는가"를 물으면 원본 물증이 소실.
- **개선 효과:** 신규 0건인 날에도 **'그날 게시판에서 무엇을 스캔했는지'(전건 목록 key)와 '원본 페이지의 무결성 지문'(SHA-256 해시)**을 **git에 영구** 남긴다. 감사자의 **일단위·장기 크롤 증거 요구**에 항구 대응.
- **리스크/대응:** 없음(순수 추가, 기존 로직 불변). 정상 응답(200)으로 스캔한 경우만 증적이 남아 "확인했다"의 신뢰성 유지.

### 🔧 기술 문서 (실무자)
- **기록 위치(`fss_crawler.js`):** 각 목록 페이지 수집 직후 `result.scanAudit`에 push — `{source, page, url, status, rowCount, bodySha256, keys[]}`. `keys`=그 페이지 전건 key, `bodySha256`=페이지 HTML의 SHA-256.
- **영구성:** `crawl_result.json`은 `.gitignore` 화이트리스트로 STEP6에서 git 강제 커밋 → scanAudit도 함께 영구 보존. `noUpdate`(신규 0)여도 기록됨.
- **구현:** `crypto` 도입, `sha256`·`openInfoKey` 헬퍼(+export). 순수 추가라 수집·판정 로직 불변. 수집 실패 시엔 성공본 미커밋이므로 scanAudit도 없음(정확).
- **감사 문답/명세:** [Q&A Q10](05_QNA.md) · [ARCHITECTURE 감사 섹션](../technical/ARCHITECTURE.md) · [기술문서 증빙](04_TECH_DOC.md).

## 건 5. 신규 판정 관측 창 차집합 전환 — `아이비케이신용정보` 미탐 사고 수정

> **건 2(게시일 앵커)를 대체한다.** 2026-07-10 설계·전환 / 2026-07-12 코드 완결·배포.

### 📋 업무 문서 (경영진)
- **배경/문제:** 금감원 제재 목록에는 **'게시일' 칸이 없다.** 목록은 **제재조치요구일 내림차순**으로 정렬되므로, 오늘 새로 올라온 건이라도 조치요구일이 지난 날짜면 목록 맨 위가 아니라 **중간에 꽂힌다.** 건 2의 게시일 앵커는 바로 그 정렬 기준에 커트오프를 건 방식이어서, **늦게 게시된 과거 조치요구일 건이 조용히 버려졌다.**
- **사고:** 실제로 **`아이비케이신용정보`**(2026-07-09 신규 게시, 조치요구일 06-25) — 레저 전체를 통틀어 **유일한 IBK 계열 건**인데 놓쳤다. 같은 방식으로 롯데손해보험·BNP파리바·순창농협·여수농협 등 12건이 함께 침묵 폐기됐다.
- **개선 효과:** 날짜가 아니라 **"직전 실행이 실제로 확인한 목록"** 과 대조해 **그 사이 새로 나타난 건만** 신규로 본다(중복방지 원장 병행). 조치요구일이 과거인 건도 새로 게시되면 정상 포착된다. 판정 근거가 없는 건(창 밖·최초 실행분)은 보고에서 빼되 **조용히 버리지 않고 기록으로 남긴다.**
- **리스크/대응:** 관측 창이 얕으면 늦게 게시된 과거 건이 창 밖에 떨어질 수 있다 → 창 깊이(기본 5페이지)가 45일치 미만이면 시스템이 **경고**한다. 폐지된 앵커(`REPORT_SINCE`)는 설정해도 **무시**되어 사고 재발이 구조적으로 차단된다.

### 🔧 기술 문서 (실무자)
- **판정 규칙(`fss_crawler.js`):** `buildScanWindow(직전 crawl_result.scanAudit)` → 소스별 `{본 key 집합, 훑은 깊이}` 복원 → `classifyRow(key, page, ledgerMap, win, seed)` 3분기.
  - 레저·직전 창에 있음 → `known`(재알림 차단) / `page ≤ win.depth` 이고 레저에 없음 → `new`(**조치요구일이 과거여도 신규**) / `seed`(최초 실행)·`page > win.depth`(창 밖) → `backfill`. 직전 창 유실 시 `depth = WINDOW_FALLBACK_DEPTH` 가정.
- **침묵 폐기 금지:** `backfill`은 `state/seen_ids.json`에만 등록하고 보고에서 제외하되 **`crawl_result.backfilled[]`에 명시 기록**한다.
- **산출/폐지:** `crawl_result.{newItems[], backfilled[], scanWindow{}, completeness{}}` + `newItemBasis: "scan-window-diff"`. `REPORT_SINCE`는 폐지(env 설정 시 무시 + 경고), `postDate` → **`actionRequestDate`** 로 필드명 정정.
- **관측 창 깊이:** `--pages` 기본 **5**(`FSS_MAX_PAGES`). `scanWindow.floorLookbackDays < 45일` → 창이 얕다고 경고.
- **수집 파이프라인 통합:** 두 소스 공통 `collectSource` — `scanSource` → `classifySnapshot` → `reconcile`(체크섬 `listTotal − prevListTotal == 신규 − 삭제`) → `buildEntry`. 체크섬 불일치·커버리지(`ledger.meta.sources[소스].covered`) 미확립 시 전 페이지 심화 스캔으로 승격.
- **슬롯 연계:** `runslot.findPreviousCrawlFile(root, dateCode, currentSlot)` — pm은 **당일 am을 직전 관측 창으로 우선** 사용.
- **검증:** 단위테스트(관측 창 차집합 판정) 통과. 2026-07-12 main 전환 런(run 29183530702) 실측 — 제재공시 21p/205행·경영유의 7p/67행 전량 훑어 커버리지 확립(`covered=true`), 신규 0 / backfill 208(과거 누적분 전량 격리, 오탐 0) → '변동 없음' 마감.

## 건 6. '목록에서 찾는 법' 참고 안내

### 📋 업무 문서 (경영진)
- **배경/문제:** 신규 건이 **목록 상단에 없다.** 목록이 제재조치요구일 순으로 정렬되기 때문에, 알림을 받은 담당자가 금감원 홈페이지에서 해당 건을 찾지 못하고 혼동한다.
- **개선 효과:** 알림·보고서 맨 끝에 **'목록에서 찾는 방법'**(제재조치요구일 · 최초 등장일 · 목록 행 번호)을 안내해, 수신자가 원문을 바로 찾아 대조할 수 있다.
- **리스크/대응:** 알림 소음 우려 → **해당 건이 없으면 안내를 아예 싣지 않는다.**

### 🔧 기술 문서 (실무자)
- **대상 판정:** `listedOutOfOrder`(제재조치요구일 < 최초 등장일)인 건에 한해 안내한다. 항목: 조치요구일 · 최초 등장일(`firstSeenDate`) · 목록 행 번호(`listRank`).
- **구현(`briefV2.js`):** `listingNoticeLines()` 공용 함수 → DOCX `buildListingNotice`(**7번째 빌더**, safeSection `"listing"`) + Telegram `tgMsg` 말미에 동일 문구 부착.

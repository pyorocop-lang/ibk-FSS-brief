## 건 1. 오전·오후 하루 2회 발화 + 오후 델타 알림

### 📋 업무 문서 (경영진)
- **배경/문제:** 기존 08:00 1회 발화만으로는, 오전 이후(전날 저녁·당일 오전) 금감원이 새로 게시한 제재를 다음 날까지 놓쳤다.
- **개선 효과:** 자매 프로젝트(FSC 법령 브리핑)와 동일하게 **08:00·16:00 하루 2회** 발화. 오후 실행은 **오전 이후 새로 게시된 건만** 알린다. 제재 대응 착수 시점이 최대 하루 앞당겨진다.
- **부담 없음:** 오후에 신규가 없으면 조용한 '변동 없음' 마감 한 줄이라 알림 피로가 없다(FSS 제재는 부정기 발행이라 대개 조용한 마감).
- **리스크/대응:** 중복 알림 우려 → 중복방지 원장으로 원천 차단(같은 건 두 번 안 감).

### 🔧 기술 문서 (실무자)
- **트리거:** Cloudflare Workers Cron에 `0 7 * * *`(=16:00 KST) 추가 → `["0 23 * * *", "0 7 * * *"]`. 대시보드에 두 cron 등록(실발화 확인). Worker는 두 cron 모두 동일하게 `workflow_dispatch` 호출.
- **슬롯:** 러너가 KST 시각으로 판별(`runslot.js`, `<12=am` / `≥12=pm`). 산출물은 `reports/{DATE}/{am|pm}/`로 분리 보존(공존·비파괴).
- **오후 델타 게이트:** `daily-brief.yml` 완료알림 스텝 — pm이고 오전본이 있으면 `notify_telegram.js --delta-since reports/{DATE}/am/crawl_result.json` 호출. 오전 이후 신규만 알림, 0건이면 '변동 없음' 마감(시작→완료 짝 보장). am이거나 오전본 없으면 평소대로 전체 전송(놓침 방지).
- **키 정정:** 델타 비교키를 `noticeId`(FSS 미존재)→`key`(examMgmtNo_emOpenSeq / 파일ID)로 수정, FSC 호환 fallback.
- **검증:** 3경로(신규有 / 0 / 오전본없음) 실코드 리허설 통과. 오늘 am·pm 자동실행 커밋으로 라이브 실증.

## 건 2. 신규 판정 정교화 — 게시일 앵커(2026-07-02) + 중복방지 레저

### 📋 업무 문서 (경영진)
- **배경/문제:** 총평단 지적 — 브리핑이 '오늘 신규'라며 게시일이 며칠~수주 전(7/2·6/26)인 제재를 함께 노출. 원인은 "우리가 전에 봤나"(레저)만으로 신규를 가려, 게시판에 누적된 **과거 공시가 신규로 샜기** 때문.
- **개선 효과:** **"게시일이 기준일(2026-07-02) 이후 AND 아직 안 본 건"만** 신규로 보고. 오래된 백로그는 조용히 배제. **'진짜 새로 게시된 것'만** 알림 → 보고 신뢰성↑, 오탐 제거.
- **기준일 근거:** 가상 시나리오 테스트로 확정 — 7/2 저녁 신규(하나은행)는 포함하고, 6/26 오래된 백로그(생보 3사)는 제외되는 최적점이 7/2.
- **리스크/대응:** 실제 신규 누락 우려 → 게시일 파싱 실패 시 fail-open(일단 보고)으로 누락 방지.

### 🔧 기술 문서 (실무자)
- **판정 규칙(`fss_crawler.js`):** 목록의 각 행이 레저에 없으면 → **레저에 등록**하되, **게시일(postDate) ≥ `REPORT_SINCE`(env, 기본 `2026-07-02`)인 건만** `newItems`/`graded`(보고 대상)로 편입. 앵커 이전 게시분(백로그)은 레저 등록 후 **상세수집·알림·보고 완전 제외**.
- **레저 병행:** `state/seen_ids.json`(성공 시에만 커밋)이 08:00·16:00 두 실행 간 중복 알림을 차단. **앵커=백로그 차단, 레저=중복 차단** — 두 축 병행.
- **기록:** `crawl_result`에 `backlogSkipped`(제외 수)·`reportSince`(앵커) 기록. `REPORT_SINCE`·`classifyTier`·`TIER_LABEL` export(시나리오 테스트용).
- **검증:** 앵커 경계(7/2·6/26→제외 / 7/3+→보고 / 빈값→fail-open) 실코드 통과. E2E 시나리오(테스트1 하나은행 보고·삼성생명 제외 / 테스트2 변동없음) 통과.

## 건 3. 사용자 노출 문구 정합 (오해 제거)

### 📋 업무 문서 (경영진)

- **개선 효과:** 각각 ① "금융감독원에서 **새로 확인된**" ② "**금융감독원** 제재·경영유의 브리핑" ③ "**공시목록 확인**"으로 정정. 수신자가 숫자·명칭·시점을 정확히 이해한다.
- **리스크/대응:** 없음(표시 문구만 변경, 판정·분석 로직 불변).

### 🔧 기술 문서 (실무자)
- **보고서 오프닝(`briefV2.js`):** "오늘 공개한"→"새로 확인된"(일반/noUpdate/빈상태 3분기 + '그 외' 헤더). 제목("오늘의 브리핑")·용어·마무리 CTA는 유지(오늘 만든 브리핑 기준이라 사실).
- **텔레그램 헤더(`briefV2`·`notify_telegram`):** "🔔 FSS…"→"🔔 금융감독원 제재·경영유의 브리핑". 코드 식별자(`fss_crawler.js`·repo명·워크플로우명·URL)는 불변.
- **pm 마감(`notify_telegram.js`):** "N건 확인"→"금융감독원 공시목록 확인"(오해 소지 숫자 제거).
- **검증 동기화:** `validator.js` C4 헤더 검사·D2 오프닝 검사를 새 문구로 정합(미동기 시 오경고). 레이아웃 정본 [`../technical/SKILL.md`](../technical/SKILL.md) 문구표 동기화.
- **적용 범위:** 사용자 노출 출력만(사용자 확정). 문서 산문·내부 로그는 불변.

## 건 4. 감사 증적 강화 — `crawl_result.scanAudit`

### 📋 업무 문서 (경영진)
- **배경/문제:** 신규가 없는 날의 원본 목록 HTML은 GitHub Artifact에 **90일만** 보관 → 감사자가 90일 초과 시점에 "그날 게시판을 실제로 확인했는가"를 물으면 원본 물증이 소실.
- **개선 효과:** 신규 0건인 날에도 **'그날 게시판에서 무엇을 스캔했는지'(전건 목록 key)와 '원본 페이지의 무결성 지문'(SHA-256 해시)**을 **git에 영구** 남긴다. 감사자의 **일단위·장기 크롤 증거 요구**에 항구 대응.
- **리스크/대응:** 없음(순수 추가, 기존 로직 불변). 정상 응답(200)으로 스캔한 경우만 증적이 남아 "확인했다"의 신뢰성 유지.

### 🔧 기술 문서 (실무자)
- **기록 위치(`fss_crawler.js`):** 각 목록 페이지 수집 직후 `result.scanAudit`에 push — `{source, page, url, status, rowCount, bodySha256, keys[]}`. `keys`=그 페이지 전건 key, `bodySha256`=페이지 HTML의 SHA-256.
- **영구성:** `crawl_result.json`은 `.gitignore` 화이트리스트로 STEP6에서 git 강제 커밋 → scanAudit도 함께 영구 보존. `noUpdate`(신규 0)여도 기록됨.
- **구현:** `crypto` 도입, `sha256`·`openInfoKey` 헬퍼(+export). 순수 추가라 수집·판정 로직 불변. 수집 실패 시엔 성공본 미커밋이므로 scanAudit도 없음(정확).
- **감사 문답/명세:** [Q&A Q10](05_QNA.md) · [ARCHITECTURE 감사 섹션](../technical/ARCHITECTURE.md) · [기술문서 증빙](04_TECH_DOC.md).

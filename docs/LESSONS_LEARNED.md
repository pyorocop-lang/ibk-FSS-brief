# 유사 아키텍처 프로젝트를 위한 기술 포인트 — 우리가 겪은 오류의 흔적과 교훈

> **누구를 위한 문서인가.** "클라우드에서 스케줄로 도는 멀티에이전트 파이프라인이, **지역이 제한된(해외에서 접근이 막히는) 데이터소스**를 긁어와, LLM으로 가공하고, **감사 추적**을 남기는" 시스템을 만들려는 사람. 이 문서는 자랑이 아니라 **우리가 실제로 밟은 지뢰의 좌표**다. 각 항목은 `증상 → 우리가 한 오진/함정 → 근본원인 → 교훈·체크리스트` 순.
>
> _작성: IBK기업은행 내부통제점검팀 · 2026-07-01 · 정본은 코드/yml, 사건 근거는 CHANGELOG·git 이력·`docs/egress_해결제안…`_

---

## 0. 이 아키텍처의 전제 (같은 함정을 밟게 되는 조건)

- **실행 위치와 데이터 위치가 다른 나라다.** 러너는 미국(GitHub Actions), 데이터는 한국 정부망. → §1
- **판단은 LLM, 나머지는 규칙.** 완전 자율이 아니라 결정론적 오케스트레이션 + 단일 LLM + 가드레일.
- **감사가 1급 요건이다.** 모든 런은 재현·추적 가능해야 하고, **기록을 덮어쓰면 안 된다.** → §4
- **1인/소수 운영 + 병행 자동 커밋.** 봇이 main에 감사 커밋을 매일 push한다. → §5, §6

이 4가지 중 2개 이상 해당하면, 아래 교훈이 거의 그대로 적용된다.

## 1. 네트워크 egress — "데이터소스 문제"가 아니라 "출구 문제"일 수 있다

**증상.** GitHub Actions(미국 러너)에서 한국 정부 API·사이트로의 요청이 **timeout**. 정시 런에서 반복 실패(한 번은 22분 52초를 헛돌다 실패).

**우리가 한 오진(2연속).**
1. **"시간대 문제"** — 아침 피크(07:30 KST)에만 실패한다고 보고 발화를 07:30→06:00으로 옮김. → 틀림. 다른 시각에도 실패(4회 중 3회). 하나의 시간 상관관계에 **과적합**했다.
2. **"한국이 해외 IP를 차단한다"** — FSC가 외국 IP를 막는다고 추정. → 틀림. 같은 API를 **다른 미국 서버(WebFetch)** 에서 부르면 정상(retMsg=200). 소스도 한국 서버도 멀쩡했다.

**근본원인.** **GitHub 러너 IP 대역 → 한국 정부망** 경로만 막힘(egress 라우팅/대역 이슈). 데이터소스(FSC↔OPEN API)를 바꿔도 동일하게 막힌다 — **소스 문제가 아니라 네트워크 출구 문제**였다.

**해결.** 한국(서울) 경유 **프록시 한 단계**. 러너 → KR 프록시(Vercel 서울 `icn1`) → 정부 API. 러너는 프록시만 부르고(미국→프록시 정상), 프록시가 서울에서 원본을 받아 **원시 응답 그대로** 반환. 결과: **22분 실패 → 57초 성공**.

**교훈·체크리스트.**
- [ ] 실패를 **데이터소스 vs 네트워크 출구**로 분리 진단하라. "같은 요청을 다른 출구에서 재현되나?"가 결정적 실험이다.
- [ ] **하나의 상관관계(시간대 등)로 결론내지 마라.** 표본을 늘려 빈도를 봐라.
- [ ] "특정 IP가 차단된다"는 가설은 **다른 IP에서의 성공/실패**로만 증명·반증하라.

## 2. 서버리스 "리전"과 "배포 모델"을 오해하면 우회가 안 되거나, 됐다가 깨진다

### 2-a. 엣지 함수는 "고정 리전"이 아니다 (Cloudflare Worker 기각)
**함정.** KR 출구를 위해 Cloudflare Worker를 먼저 시도. → **호출자 근처(미국 SJC)에서 실행**되어 서울 고정이 불가. 엣지=사용자 근처지 원하는 나라가 아니다.
**교훈.** "서버리스로 한국에서 실행"하려면 **리전 고정이 가능한 플랫폼**(Vercel `regions:["icn1"]`, AWS Lambda 서울 등)을 골라라. `region` 필드를 응답에 실어 **실제 실행 위치를 측정**하라(우리는 `X-Proxy-Region: icn1`로 검증).

### 2-b. Git 연동 + Root Directory 오설정 = 매 푸시가 프로덕션을 덮어쓴다 (가장 값비싼 사건)
**증상.** 잘 돌던 프록시가 어느 순간 **apex 포함 전 경로 HTTP 404**. 재배포하면 잠깐 살았다가 **다음 날 또 죽음**.

**우리가 한 오진.** "Vercel Hobby 배포가 만료/소멸했다"(원인 불명 SPOF)로만 기록. → 원인을 못 짚어 재배포=반창고에 그칠 뻔했다.

**근본원인(확정).** Vercel 프로젝트가 저장소 `main`에 **Git 연동**돼 있고, **Root Directory가 저장소 루트**로 설정됨. 프록시 함수는 하위폴더 `vercel-kr-crawl/api/`에 있는데 루트엔 `api/`가 없어, **매 `main` 푸시가 함수 없는 빈 배포(빌드 2초)를 프로덕션에 찍어** 정상 배포를 덮어씀. **이 저장소는 매일 감사 커밋을 push**하므로 하루 내 재발.
- **확정 근거(배제법):** 배포 목록에서 `main` git 커밋 배포는 전부 404(빌드 2s), `vercel-kr-crawl/` 폴더로 올린 CLI 배포(빌드 5s)만 정상. 401/403/502/DNS/토큰 전부 배제.

**해결.** Root Directory를 `vercel-kr-crawl`로 지정. 이후 모든 push가 그 폴더를 빌드 → 프록시가 올바로 재배포. push가 더는 프록시를 깨지 않음(git 재배포에서도 200·icn1 재확인).

**교훈·체크리스트.**
- [ ] **PaaS의 "배포 트리거"를 정확히 이해하라.** git 연동이면 **모든 push가 재배포**다. 하위폴더 프로젝트면 **Root Directory**를 반드시 그 폴더로.
- [ ] **CLI 배포와 git 연동 배포는 서로 다른 소스에서 빌드**한다(로컬 디렉터리 vs 대시보드 설정). "CLI로 됐는데 왜 죽지?"의 답이 여기 있다.
- [ ] 인프라 설정(Root Directory 등)을 **운영 정본 문서에 명문화**하라(config-as-ops). 우리는 `vercel-kr-crawl/README.md`에 "Root Directory=필수"를 박아뒀다.
- [ ] 404가 **apex까지** 나면 "라우트 문제"가 아니라 "**배포 자체가 빈 것**"을 의심하라(401=인증, 403=allowlist, 502=업스트림과 구분).

## 3. 실패를 잘못 기록하면, 진짜 원인이 몇 주씩 숨는다 (오진 유발 설계)

**증상.** 수집 실패의 `error`가 `"timeout"`으로 기록됨. 그런데 실제 1차 원인은 **프록시 404**였다.

**근본원인(연쇄).** 1차 OPEN API가 프록시 404로 즉시 실패 → **FSC 스크래핑 fallback이 프록시를 안 타고 직결** → 러너→정부망 egress로 timeout → 최종 `error:"timeout", source:FSC`로 기록. **1차 원인(프록시 404)이 fallback의 timeout에 가려짐.** 게다가 90s × 3재시도 × 3워크플로시도 = **약 18분 헛돎**.

**해결.**
- **① fallback도 같은 우회 경로로.** FSC fallback도 `viaProxy` 경유(직결 egress 회피).
- **② 헬스체크 + fail-fast.** 프록시가 죽었으면 doomed한 루프를 돌지 말고 **즉시 실패**(18분 → 4초). timeout 90s→30s, 재시도 간격 120s→60s.
- **③ 실패 사유 분리 기록.** `failure_meta.json`에 `apiError`·`fallbackError`·`proxy`(헬스)를 분리 보존하고, `error`를 복합 문자열(예: `프록시:404 / API:HTTP 404`)로 → 알림·요약에도 **진짜 원인** 노출.

**교훈·체크리스트.**
- [ ] **다단 fallback은 "1차 실패 사유"를 삼키지 않게** 각 단계 사유를 따로 남겨라. 최종 에러 한 줄이 진실을 가린다.
- [ ] **선행 의존성(프록시 등)은 헬스체크로 먼저 확인**하고, 죽었으면 하위 재시도를 **fail-fast**로 끊어라. 재시도 폭풍은 시간·비용·오진을 부른다.
- [ ] fallback이 **주 경로와 다른 네트워크 특성**을 갖지 않는지 봐라(우리 fallback은 프록시를 안 타서 새 실패 모드를 만들었다).

## 4. 기록은 절대 덮어쓰지 마라 — 실패가 성공을 지우면 감사가 무너진다

**증상(과거).** timeout으로 실패한 런이 **성공했던 수집 6건을 0건으로 덮어씀**. 감사 관점에서 치명적.

**근본원인.** 실패 시에도 같은 산출물 파일(`crawl_result.json`)을 갱신하는 설계.

**해결(절대규칙).**
- 수집 실패 시 **성공본을 건드리지 않고** `failure_meta.json`만 별도 기록 → 커밋 단계가 그 신호로 **성공 데이터/실패 로그를 분리**.
- 런별 산출물을 `reports/{날짜}/{am|pm}/`에 **비파괴 분리 보존**(덮어쓰기 금지).
- 상태는 **전부 성공해야만 "정상"**(all-or-nothing). "부분 성공"으로 뭉개지 않는다.

**교훈·체크리스트.**
- [ ] **실패 경로가 성공 산출물을 만지지 못하게** 물리적으로 분리하라(다른 파일/폴더).
- [ ] 각 런은 **독립적으로 공존**해야 한다(슬롯/타임스탬프 폴더). "최신만 남기기"는 감사에서 독이다.
- [ ] **"모르면 멈춘다."** 수집 실패를 "해당 없음"으로 오인 보고하지 마라 — 데이터 미확인 ≠ 안전. 명시적 실패 + 알림 + 재실행 유도.

## 5. Windows + git 개발환경의 지뢰 (재현·복구에 시간을 가장 많이 뺏김)

**증상 1 — 파일 잠금.** `git pull` 중 `unable to unlink old '….yml': Invalid argument`. 에디터/백신이 파일 핸들을 잡고 있어 git이 **파일을 교체(unlink)하지 못해** 병합이 중단.

**증상 2 — 경로 변환 사고.** 복구하려고 `git show origin/main:path > file` 했더니 MSYS(Git-Bash)가 콜론 인자를 Windows 경로로 오해해 `origin/main:path` → `origin\main;path`로 **뭉갬** → git이 에러를 내고 그 **에러 메시지가 파일에 기록**되어 워크플로 파일이 손상됨.

**핵심 발견.** 잠긴 파일도 **truncate-쓰기(제자리 덮어쓰기)는 되고, unlink만 막힌다.** 이걸 이용해 복구·우회.

**해결.**
- 손상 파일을 `MSYS_NO_PATHCONV=1 git show "rev:path" > file`로 **제자리 복구**(경로변환 차단).
- FF가 unlink로 막히면: 대상 파일을 origin 내용으로 제자리 쓰기 → `git add`로 인덱스를 target과 일치 → `git merge --ff-only`가 "이미 최신"으로 판단해 **unlink를 건너뜀** → 병합 성공.

**교훈·체크리스트.**
- [ ] Windows에서 git 작업 전 **대상 파일을 에디터에서 닫아라**(특히 pull/checkout이 건드릴 파일).
- [ ] Git-Bash에서 `rev:path`처럼 **콜론 든 인자**는 `MSYS_NO_PATHCONV=1`로 감싸라.
- [ ] `cmd > file` 리다이렉트는 **명령이 실패해도 파일을 truncate**한다 — 실패 시 파일이 에러문으로 오염될 수 있다. 중요한 파일엔 임시파일→검증→이동을 써라.

## 6. 병행 자동 커밋 · staged 파일 보존 (1인 저장소라도 "협업"이다)

**상황.** 봇이 매일 감사 커밋을 `main`에 push → 로컬이 수시로 **behind/divergence**. 게다가 사용자가 **staged 상태로 둔 파일**(작업 중 산출물)이 있어, 함부로 `reset --hard`/전체 `commit`하면 날아간다.

**해결.**
- 동기화는 **stash → pull --ff-only → (필요시 우회) → pop** 또는 파일 잠금 우회(§5).
- 커밋은 **명시적 pathspec**(`git commit <내 파일들> -m …`)으로 — 사용자의 staged 파일은 인덱스에 그대로 남는다.

**교훈·체크리스트.**
- [ ] **자동 커밋 주체(봇)가 있으면 항상 rebase/ff를 전제**로 로컬 작업하라.
- [ ] 남의(또는 미래의 나의) **staged 변경을 내 커밋에 휩쓸지 마라** — pathspec 커밋으로 범위를 좁혀라.
- [ ] `reset --hard`는 staged 신규 파일을 애매하게 처리한다 — 감사 산출물엔 쓰지 마라.

## 7. 사양 ↔ 구현 드리프트, 그리고 문서 정합성

**증상.** 보고서 "사양"은 옛 형식(❶~❺ 고정번호·표)인데 **실제 코드 출력은 뉴스레터형**(🌞/🔴/🔹/📖). 검증기(validator)는 **보고서 구조를 전혀 검사하지 않아** 드리프트를 못 잡음. 또 파일명 슬롯 버그(pm인데 항상 `morning`)가 조용히 존재.

**해결.**
- **실제 코드 출력을 정답으로** 사양(SKILL.md)·규칙(CLAUDE.md)을 현행화.
- validator에 **구조 검증(D)** 신설 — 데이터로 기대 섹션을 계산해 실제 docx와 대조(사양↔구현 드리프트 자동 포착).
- 회귀 테스트로 슬롯 버그를 잠금(오프라인 픽스처).

**교훈·체크리스트.**
- [ ] **"문서가 말하는 것"과 "코드가 하는 것"을 주기적으로 대조**하고, 충돌 시 **코드를 정본**으로 삼아 문서를 고쳐라.
- [ ] 산출물 구조를 **자동 검증**하라(사람 눈은 드리프트를 놓친다).
- [ ] 같은 사실을 여러 문서에 복붙하지 마라 — **단일 출처 + 포인터 stub**. 코드/동작을 바꾸면 **그 사실을 언급한 모든 문서를 같은 커밋에서** 갱신(정합성 규율).

## 8. 진단 방법론 — "사실을 다시 확인하라"의 반복 (오진의 메타 교훈)

이 프로젝트의 오진들은 공통 패턴이 있었다: **로컬/부분 관찰을 전체로 성급히 확장.**
- 로컬 env의 **Telegram 토큰 401**(옛 토큰 revoke) → "운영 봇이 죽었다"로 오판. 실제 운영(GitHub Secret)은 정상. → **토큰은 만료되지 않는다; 401=무효/폐기; 로컬≠운영.**
- "timeout"이라는 최종 에러 → 진짜 원인(프록시 404)을 가림(§3).
- "시간대"·"해외 IP 차단" 가설(§1).

**교훈·체크리스트.**
- [ ] 결론 전에 **"이 관찰이 전체를 대표하나? 다른 환경에서 재현되나?"**를 물어라.
- [ ] **증상(문자열)이 아니라 계층(HTTP 코드·리전·경로)** 으로 원인을 좁혀라(404/401/403/502/000은 각각 다른 이야기다).
- [ ] 사용자가 "그럴 리 없다"고 하면 **가설을 재검증**하라 — 우리의 여러 오진은 그 지점에서 바로잡혔다.

## 9. 스케줄·트리거 — 정시성은 별도 관심사

**교훈.** GitHub 자체 `schedule` cron은 **지연·누락이 잦다**. 정시성이 중요하면 **외부 트리거**(우리는 Cloudflare Workers Cron → `workflow_dispatch`)로 분리하라. 스케줄링과 실행을 한 곳에 묶지 마라.

## 10. 한 장 요약 — 프로젝트 시작 전 체크리스트

- [ ] 러너와 데이터가 **다른 나라**인가? → egress를 **소스 문제와 분리**해 먼저 검증(§1). 우회는 **리전 고정 가능한** 플랫폼으로(§2-a).
- [ ] 우회 인프라가 **git 연동 PaaS**인가? → **Root Directory·배포 트리거**를 정확히 잡고 **운영 정본에 명문화**(§2-b).
- [ ] 실패를 **계층·사유별로 분리 기록**하고, 선행 의존성은 **헬스체크+fail-fast**(§3).
- [ ] **실패가 성공 기록을 못 덮게**, 런별 **비파괴 분리 보존**, **all-or-nothing**, **"모르면 멈춘다"**(§4).
- [ ] Windows면 **파일 잠금·MSYS 경로변환**을 각오(§5). 봇 자동 커밋이면 **ff/pathspec**로 안전 커밋(§6).
- [ ] **사양↔구현 드리프트**를 자동 검증하고, **문서 단일 출처 + 정합성 규율**(§7).
- [ ] 오진 방지: **로컬≠운영**, 증상 아닌 **계층으로** 진단(§8). 정시성은 **외부 트리거**로(§9).

# AI 내부통제 모니터링 시스템 — 경영진 보고

**보고 대상:** 내부통제총괄부 / C-Level  
**보고 일자:** 2026년 6월 30일 (현행화 — 완전 클라우드 이전·정시 2회·egress 과제 반영)  
**작성:** 내부통제점검팀

---

## 1. 한 줄 결론

> 금융위원회 입법예고를 매일 자동으로 수집·분석해  
> **담당 부서별 실무 액션이 담긴 보고서를 오전 8시 전에 자동 생성**합니다.  
> 현재 운영 중이며, 인력 투입 없이 매 영업일 실행됩니다.

---

## 2. 왜 만들었는가

### 기존 방식의 문제

| 항목 | 기존 수동 방식 |
|---|---|
| 소요 시간 | 금융위원회 홈페이지 접속 → 목록 확인 → 개별 열람 → 요약 작성 → 부서 배분 (약 1~2시간/일) |
| 누락 위험 | 담당자 부재·휴가 시 모니터링 공백 발생 |
| 일관성 | 담당자마다 중요도 판단·보고 형식이 달라짐 |
| 대응 속도 | 수집 후 검토까지 당일 오전 중반 이후 |

### AI 시스템 도입 후

| 항목 | 현재 자동화 방식 |
|---|---|
| 소요 시간 | 오전 06:00 자동 시작 → 수분 내 보고서 완성 (무인) · 오후 16:00 추가 모니터링 |
| 누락 위험 | 시스템 자동 실행 — 담당자 부재와 무관 |
| 일관성 | AI가 동일한 기준으로 분석, 표준 포맷 유지 |
| 대응 속도 | 출근 전 보고서 완성, 오전 업무 시작 즉시 검토 가능 |

## 3. 시스템이 하는 일 (5단계)

```
06:00  ① 수집     금융위원회 소관 진행중 입법·행정 예고 수집 (클라우드, 자동)
                   → 1차: 정부입법지원센터 OPEN API / 2차: FSC 스크래핑 fallback
                   → 원문 포함 저장 (감사 대응)

      ② 분석     Claude AI가 IBK 영향도 평가
                   → 🔴 즉시조치 / 🔶 중간검토 / 🔹 참고 등급 분류
                   → 담당 부서 자동 배정 (10개 부서)

      ③ 보고서   DOCX 보고서 자동 생성
                   → 5개 섹션: 요약 / 법령브리핑 / 체크포인트 / 마감캘린더 / 용어해설

      ④ 검증     AI 출력 품질 자동 검증 (10개 항목)
                   → 부서명 오류·글자 수 초과·형식 불일치 자동 감지

      ⑤ 알림     Telegram으로 완료 알림 + 핵심 내용 즉시 전송
                   → 담당자가 이동 중에도 즉시 확인 가능

16:00  오후 추가 모니터링 (pm) — 공무원 근무시간 게시분 반영
                   → 완료 알림 항상 전송(델타): 신규 있으면 전체 알림, 없으면 "오전 대비 변동 없음" 마감. 보고서는 항상 생성·보존
```

## 6. 운영 구조 (한 눈에)

```
[전 과정 클라우드 — 로컬 PC 불필요]
  ⏰ Cloudflare Workers Cron (06:00 am / 16:00 pm KST)
         │  workflow_dispatch
         ▼
  ☁️ GitHub Actions 단일 Job
  ┌──────────────────────────────────────┐
  │ ① 수집  정부입법지원센터 OPEN API 1차   │
  │         + FSC 스크래핑 fallback        │
  │ ② 분석  Claude AI (IBK 영향도·부서배정) │
  │ ③ 보고서 DOCX 생성                     │
  │ ④ 검증  품질 자동 점검                  │
  │ ⑤ 아카이브·감사 커밋                    │
  └──────────────────────────────────────┘
         │
         ▼
  [결과물 자동 저장]
  • Telegram 알림 → 담당자 스마트폰 (am 항상 / pm 델타)
  • DOCX 보고서  → GitHub Artifact 90일 보관 (런별 슬롯 분리)
  • 원문·수집데이터 → git 커밋(감사 추적)
```

> **PC가 꺼져 있어도** 전 과정이 클라우드(GitHub Actions)에서 자동 실행됩니다. 로컬 PC는 필요 없습니다.  
> ⚠️ **과거 제약(해결됨):** GitHub 러너(미국 소재) IP에서 한국 정부망(FSC·OPEN API 공통)으로의 직결이
> timeout(egress)됩니다. 과거 "FSC가 해외 IP를 차단한다"는 진단은 정확하지 않았고, 본질은
> 러너→정부망 egress입니다. **해결책으로 KR(한국) 경유 프록시(Vercel 서울 icn1)를 도입·운영 검증 완료(2026-06-30)** 했습니다 —
> 러너→프록시→정부 API 경로로 정시 런이 57초에 정상 성공합니다(분석·보고서·알림 포함 전 구간). 장기 안정화(고정 IP)는 후속.

## 8. 현재 상태 및 다음 단계

### 현재 상태 ✅ 정상 운영 중 (egress 해결 완료)

- 매 영업일 오전 06:00(am)·오후 16:00(pm) 클라우드 자동 실행 (로컬 PC 불필요)
- 데이터소스 전환 완료: FSC 스크래핑 → 정부입법지원센터 OPEN API 1차(스크래핑 fallback)
- **egress 해결·운영화(2026-06-30):** KR 경유 프록시(Vercel 서울 icn1)로 러너→정부 API 도달 — 정시 런 57초 정상 성공. 프록시는 정식 운영 배포로 관리(자동 재배포).
- **회복력 입증(2026-07-01):** 프록시에 배포 설정 결함으로 장애가 발생했으나, 자동 감시가 즉시 실패로 포착 → 근본원인 규명 → **재발방지 완료**(오보 없이 안전하게 처리).
- Telegram 알림(am 항상/pm 델타) + DOCX 보고서 자동 생성, 런별 슬롯 분리 보존
- 원문·수집데이터 감사 보관 적용
- 후속(선택): 장기 가용성 강화를 위한 KR 고정 IP + OPEN API 공인 IP 등록(제안서 §5-B)

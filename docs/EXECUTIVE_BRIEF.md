# AI 제재·경영유의 모니터링 시스템 — 경영진 보고

**보고 대상:** 내부통제총괄부 / C-Level  
**보고 일자:** 2026년 7월 2일 (FSS 현행 구현 기준 갱신)  
**작성:** 내부통제점검팀

---

## 1. 한 줄 결론

> 금융감독원(FSS)이 공개하는 **제재공시·경영유의사항**을 매일 자동으로 수집·분석해  
> **IBK 자가점검·벤치마킹 관점의 실무 액션이 담긴 보고서를 오전 8시대에 자동 생성**합니다.  
> 인력 투입 없이 매일 클라우드에서 실행됩니다.

> **성격:** 사후(事後) 모니터링입니다. 타행·인접 금융회사의 실제 제재사례에서 IBK가 유사 취약점을
> 갖고 있는지 자가점검하고, IBK가 직접 제재 대상이면 즉시 대응하도록 돕습니다.
> (법령 시행 전 예방 목적의 입법예고 브리핑과는 목적·소스가 다릅니다.)

---

## 2. 왜 만들었는가

### 기존 방식의 문제

| 항목 | 기존 수동 방식 |
|---|---|
| 소요 시간 | 금감원 홈페이지 접속 → 제재공시·경영유의 목록 확인 → 개별 열람·PDF 다운로드 → 요약 → 부서 배분 |
| 누락 위험 | 공시가 부정기적이라 매일 확인하지 않으면 신규 제재를 놓침 |
| 일관성 | 담당자마다 IBK 연관성·중요도 판단 기준이 다름 |
| 대응 속도 | 타행 제재사례를 IBK 자가점검으로 연결하기까지 지연 |

### AI 시스템 도입 후

| 항목 | 현재 자동화 방식 |
|---|---|
| 소요 시간 | 오전 08:00대 자동 시작 → 수분 내 보고서 완성 (무인) |
| 누락 위험 | 매일 실행 + 중복방지 원장(seen_ids)으로 신규분만 정확히 포착 |
| 일관성 | AI가 동일한 Tier·위험도 기준으로 분석, 표준 포맷 유지 |
| 대응 속도 | 출근 전 보고서 완성, 유사 업무 점검 제안을 즉시 관련 부서로 |

## 3. 시스템이 하는 일 (5단계)

```
08:00대  ① 수집     금융감독원 신규 제재공시·경영유의 수집 (클라우드, 자동)
                    → 2소스: 제재공시(openInfo, HTML+PDF) · 경영유의(openInfoImpr, PDF)
                    → 중복방지 원장(seen_ids)과 대조해 신규분만 선별
                    → 원문 PDF 포함 저장 (감사 대응)

        ② 분석     Claude AI가 IBK 벤치마킹 관점으로 신규건만 분석
                    → 기관 계층 Tier(T0 IBK / T1 은행 / T2 인접금융 / T3 주변) 분류
                    → 위험도 상/중/하 판정 + 담당 부서 자동 배정
                    → 타행 제재는 "IBK 자가점검 제안", IBK 직접 제재는 "즉시 대응"

        ③ 보고서   DOCX 보고서 자동 생성
                    → 요약 / 제재대상 카드(무슨 일→IBK 발생 가능?→점검) / 그 외 목록 / 오늘의 용어

        ④ 검증     AI 출력 품질 자동 검증
                    → 부서명 오류·글자 수 초과·형식 불일치 자동 감지

        ⑤ 알림     Telegram으로 완료 알림 + 핵심 제재사례 즉시 전송
                    → 담당자가 이동 중에도 즉시 확인 가능
```

> 제재는 부정기적으로 공시되므로 **오전 1회(단일 슬롯)** 실행이 기본입니다.
> 필요 시 수동 재실행하면 오후(pm) 슬롯으로 분리 저장되어 오전 기록을 덮지 않습니다.

## 5. 커버리지 및 정확도

| 항목 | 현황 |
|---|---|
| 모니터링 대상 | 금융감독원 제재공시 + 경영유의·개선사항 신규 전건 |
| 중요도 체계 | 기관 계층 Tier(T0/T1/T2/T3) × 위험도(상/중/하) — 제재 마감(D-day) 개념 없음 |
| 알림 노출 | T0·T1·T2 전건 (IBK 무관 소규모 T3 제외, 건수는 헤더 표기) |
| 보고서 수록 | 전건 포함 (T3도 하위 수록) |
| 부서 자동 배정 | IBK 공식 조직도 기준 (자금세탁방지부·정보보호총괄부·금융소비자보호부 등) |
| 분석 모델 | Anthropic Claude Haiku 4.5 (claude-haiku-4-5) |
| PDF 원문 보관 | 감사 대응용 원문 GitHub Artifact 저장 (90일) · 런별 슬롯 분리 보존 |
| 해외 IP 접근 | FSS는 해외 IP 차단 없음이 검증됨(미국 러너 접근 PASS) → 프록시 불필요 |

## 6. 운영 구조 (한 눈에)

```
[전 과정 클라우드 — 로컬 PC 불필요]
  ⏰ Cloudflare Workers Cron ("0 23 * * *" = 08:00 KST 매일)
         │  workflow_dispatch
         ▼
  ☁️ GitHub Actions 단일 Job  ("IBK FSS Sanction Brief")
  ┌──────────────────────────────────────────┐
  │ ① 수집   FSS 2소스 스크래핑 + seen_ids dedup │
  │ ② 분석   Claude AI (Tier·위험도·부서배정)    │
  │ ③ 보고서 DOCX 생성                          │
  │ ④ 검증   품질 자동 점검                      │
  │ ⑤ 아카이브·감사 커밋                         │
  └──────────────────────────────────────────┘
         │
         ▼
  [결과물 자동 저장]
  • Telegram 알림 → 담당자 스마트폰
  • DOCX 보고서  → GitHub Artifact 90일 보관 (런별 슬롯 분리)
  • 원문 PDF·수집데이터·seen_ids 원장 → git 커밋(감사 추적)
```

> **PC가 꺼져 있어도** 전 과정이 클라우드(GitHub Actions)에서 자동 실행됩니다. 로컬 PC는 필요 없습니다.  
> GitHub 자체 schedule cron은 지연·누락이 잦아, 정시성은 Cloudflare Workers Cron이 책임집니다.

## 8. 현재 상태 및 다음 단계

### 현재 상태 ✅ 정상 운영

- 매일 오전 08:00대 클라우드 자동 실행 (로컬 PC 불필요)
- 수집: FSS 2소스(제재공시 HTML+PDF / 경영유의 PDF) 스크래핑 + seen_ids 중복방지
- FSS 해외 IP 차단 없음 검증 완료 → KR 경유 프록시 불필요 (순수 스크래핑)
- 수집 실패를 "신규 없음"으로 오인 보고하지 않도록 실패 격리(failure_meta) 적용
- Telegram 알림 + DOCX 보고서 자동 생성, 런별 슬롯 분리 보존
- 원문 PDF·수집데이터·seen_ids 원장 감사 보관 적용

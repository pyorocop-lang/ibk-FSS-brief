## 0. 한눈에 보는 전체 흐름

법령 이슈 1건이 "담당 부서 + 실무 액션"으로 바뀌기까지는 **3단계**를 거칩니다.
각 단계는 서로 다른 방식(규칙 기반 / AI 추론 / 규칙 기반 백업)으로 동작하며, 모두 **공개·공식 출처에 근거**합니다.

```
[1단계] 수집·선별·등급             [2단계] 부서·액션 배정             [3단계] 보고서·알림
 (규칙 기반, 사람이 검증 가능)       (AI 추론 + 공식 지식 주입)          (템플릿)
─────────────────────────         ─────────────────────────        ──────────────
 금융위 입법·행정예고 수집        →  Claude Haiku LLM 분석          →  docx 보고서
 키워드 사전으로 IBK 관련 필터       (4개 지식파일을 근거로 주입)        Telegram 알림
 점수 합산 → 상/중/하 등급           → dept(주담당)·related_depts        
                                     our_action(실무 액션) 산출       
 [fsc_crawler.js]                   [analyst.js + knowledge/*.md]     [briefV2.js]
                                          │
                                          └─(AI 호출 실패 시)→ 키워드 규칙 백업(fallback)
```

비유: **병원 분류(triage)** 와 같습니다.
- 1단계 = 접수처에서 증상 키워드로 "응급/일반/경증"을 나눔(규칙).
- 2단계 = 전문의가 진료지침서를 펴 놓고 "어느 과(부서)가 맡을지" 판단(AI + 지침).
- 진료지침서(공식 내규·조직도)가 곧 **데이터 원천**입니다.

## 질문 1. LLM이 이슈와 부서를 어떻게 매핑하는가

### 1-1. 매핑은 "AI 단독 추측"이 아니라 "공식 지식을 펴 놓고 하는 추론"

핵심 원리는 **RAG(검색 증강 생성)에 가까운 지식 주입 방식**입니다.
AI가 학습된 기억에만 의존해 부서명을 지어내지 않도록, 매번 분석 직전에 **IBK 공식 문서 4종을 시스템 프롬프트에 통째로 붙여 넣습니다.**

근거 코드 — [analyst.js:62-74](../analyst.js#L62-L74):
```
K_DEPT    = ibk-dept-mapping.md   (조직도 기반 부서 매핑)
   K_ORG     = ibk_org_chart.md      (15그룹·1연구소·4본부·1단·2센터·75부)
K_MAPPING = ibk_mapping_rules.md  (공식 내규 목록 PDF)
K_ACTION  = ibk_action_rules.md   (공개 모범규준 기반 추론 지침)
→ SYSTEM_PROMPT = 기본 지시문 + 위 4개 파일을 런타임에 동적 결합
```
즉 주된 매핑 기준은 `knowledge` 폴더의 Markdown에서 갱신됩니다. 다만 `analyst.js` fallback이나 시스템 프롬프트처럼 부서명이 하드코딩된 부분은 별도 정합성 점검이 필요합니다.

### 1-2. AI가 따르는 배정 순서 (결정적 → 추론적)

AI는 다음 우선순위로 부서를 특정합니다. 위쪽일수록 "정해진 표를 그대로 따르는" 결정적 방식이고, 아래로 갈수록 "지침에 근거한 추론"입니다.

1. **법령 영역 → 업무분야 → 소관부서 직접 매칭**
   `ibk_mapping_rules.md`의 ①14개 업무분야 참조표 / ②법령영역별 영향내규 테이블에서 가장 가까운 행을 찾아 **주담당 부서(dept)** 를 그대로 가져옵니다.
   예: "자금세탁·특금법·FIU" → 자금세탁방지부 / "개인·신용정보" → 정보보호총괄부.

2. **테이블에 없으면 공개 모범규준으로 추론**
   `ibk_action_rules.md`가 정한 순서대로 참조 — ①금융위「금융회사 지배구조 모범규준」 ②금감원「내부통제 모범규준」 ③법령별 금융업권 적용 가이드라인.
   예: 신용정보법 → "신용정보관리·보호인 지정 의무" → 정보보호총괄부.

3. **다중 부서 분리 규칙**
   - `dept`: 가장 직접적으로 업무가 바뀌는 **주담당 1곳**(규정 개정 의무가 있는 내규의 소관부서).
   - `related_depts`: 준수·협조만 하는 부서. **반드시 매핑 테이블에 존재하는 부서만**, 최대 2개. 근거 없으면 빈 배열.
     ([analyst_system_prompt.md:91-98](../agents/analyst_system_prompt.md#L91-L98))

4. **조직개편 불확실성 통제**
   - 직제자료는 조직의 명칭과 소속을 확인하는 근거로 사용하고, 개정 전후 대비표의 직접 명칭변경·소속이동은 현행 명칭과 소속으로 반영한다.
   - 세부 업무소관은 「업무분장규정」 또는 별도로 기록된 명시적 업무승계 확인으로 확정한다.
   - 2026-07-15 사용자 확인에 따른 데이터혁신·WM·브랜드전략·BCP/안전 업무 승계를 적용한다.
   - 대비표에서 삭제와 신설이 별도로 표시되었거나 과거 명칭 자체가 확인되지 않으면 유사한 신설부서로 자동 승계하지 않고 `소관부서 확인 필요`로 남긴다.

### 1-3. "지어낸 부서명"을 막는 가드레일

설명 가능성·정확성을 위해 **존재하지 않는 부서명을 출력하지 못하도록 이중 차단**합니다.
- 금지 목록 명시: 법무실, AML팀, 개인정보보호팀, IT운영부 등 → 반드시 정식 부서명으로 대체. `경영전략부`는 2026-07-14 직제상 공식 부서이므로 금지하지 않는다.
- 출력은 **자유 문장이 아니라 정해진 JSON 형식**(dept / related_depts / our_action / ctrl_insight …)으로 강제한다. 다만 현재 `validator.js`에는 조직 정본과 대조하는 부서명 화이트리스트 검사가 구현되어 있지 않으므로 별도 정합성 검사가 필요하다.

### 1-4. 액션 룰(our_action) 산출 방식

부서가 정해지면, 같은 AI 호출에서 **"그 부서 담당자가 오늘 할 일"** 을 함께 만듭니다.
- 형식 고정: `"[부서명] 담당자라면 ~꼭 확인해 보세요"` (제안형, 60자 이하)
- 우선순위: ①시행일/의견마감 기한이 있으면 기한 기준 액션 ②등급 상·중이면 주제별 구체 액션
- 주담당 액션 1개 필수 + 협조부서 액션 최대 2개 = 총 1~3개
  ([analyst_system_prompt.md:118-121](../agents/analyst_system_prompt.md#L118-L121))

## 질문 2. 적용된 기술은 무엇인가

| 구성요소 | 적용 기술 | 역할 | 근거 |
|---|---|---|---|
| 1차 선별·등급 | **규칙 기반 점수 산정** (키워드 사전 + 가중치) | IBK 관련성 필터, 상/중/하 등급 | `fsc_crawler.js` + `ibk-keywords.md` |
| 부서·액션 배정 | **LLM (Claude Haiku 4.5)** — 모델 ID `claude-haiku-4-5-20251001` | 법령→부서→액션 추론 | [analyst.js:40](../analyst.js#L40) |
| 지식 연결 | **지식 주입(RAG 방식)** — 공식 .md 4종을 시스템 프롬프트에 동적 결합 | AI가 공식 근거만 보고 판단 | [analyst.js:62-74](../analyst.js#L62-L74) |
| 출력 신뢰성 | **구조화 JSON 강제** + 형식·글자수 제약 | 기계 검증 가능한 결과 | `analyst_system_prompt.md` |
| 호출 안정성 | **재시도 로직** (timeout·429·5xx 시 1s→3s→9s 지수 백오프, 최대 3회) | 일시적 API 오류 흡수 | [analyst.js:130-149](../analyst.js#L130-L149) |
| 무중단 설계 | **Fallback(규칙 기반 백업)** — API 키 없거나 오류 시 키워드 매핑으로 자동 전환 | AI 없이도 결과 보장 | [analyst.js:152-274](../analyst.js#L152-L274) |
| 품질 검증 | **validator.js** — 문체·필수필드·알림형식 사후 검증 | 형식·문체 오류 탐지. 부서명 정본 대조는 현재 미구현 | `validator.js` |

### 핵심 기술 포인트 3가지 (감사 설명용)

1. **AI는 "기억"이 아니라 "제출된 공식 문서"로 판단한다.**
   매 호출마다 IBK 조직도·내규목록·모범규준을 근거 자료로 함께 제출한다. 사용된 지식파일 집합은 확인할 수 있지만, 개별 부서 판단 근거는 현재 최종 JSON에 별도 필드로 남지 않는다.

2. **AI가 실패해도 멈추지 않는다(이중화).**
   API 장애 시 동일한 키워드 규칙(`fallbackAnalyze`)으로 자동 전환되어 부서·액션을 산출.
   단, 이 경우 결과에 `_fallback` 표시가 남아 **AI판/규칙판을 사후 구분**할 수 있음.

3. **모든 출력은 사람이 검증 가능한 형태다.**
   자유 서술이 아닌 JSON·고정 형식으로 남는다. 현재 validator는 글자수·금지표현·필수필드를 점검하지만, 부서명 존재 여부는 별도 검증이 필요하다.

> 참고: 부서 배정 추론의 "근거"는 최종 JSON에 별도 필드로 노출하지 않습니다([ibk_action_rules.md:8-9](ibk_action_rules.md#L8-L9)). 다만 어떤 지식 파일을 근거로 삼았는지는 본 문서의 매핑 테이블로 역추적 가능합니다.

### 3-3. 데이터 신뢰성·갱신 원칙

- **출처 명시 의무**: 조직·부서 매핑 지식파일 상단에 조직 기준자료, 업무 매핑 원천, 갱신일과 미확정 범위를 구분해 기록한다.
- **공식 문서 우선 정정 이력 보존**: 파일명 기반 추정 부서를 공식 PDF로 교체한 이력을 문서에 남김(예: 개인·신용정보 규정 소관을 준법지원부 → **정보보호총괄부**로 정정).
- **코드와 지식의 분리 원칙**: 기본 매핑은 지식파일에서 관리한다. 다만 현재 `analyst.js` fallback과 시스템 프롬프트에는 일부 부서명이 하드코딩되어 있어 조직개편 때 코드 정합성도 함께 점검해야 한다.
- **권한 있는 근거의 분리**: 직제 대비표의 직접 명칭변경·소속이동과 별도의 명시적 업무승계 확인을 구분해 기록하며, 어느 쪽도 없는 삭제·신설 조직 관계는 추정하지 않음.

## 부록 A. 등급 판정(상/중/하) 산식 — 1단계 규칙의 투명성

부서 배정 전에, 어떤 이슈를 다룰지 정하는 등급은 **완전히 결정적인 산식**입니다(AI 개입 없음, 100% 재현 가능).

```
점수 = (Tier1 키워드 매칭 ? +3) + (Tier2 키워드 매칭 ? +1)
       + 마감 임박 보정 (D-14 이내 +2 / D-15~30 +1)

4점 이상 → 상 🔴 (즉시 검토)
2~3점   → 중 🔶 (관심 모니터링)
1점     → 하 🔹 (참고)
0점     → 무관 (보고서 제외)
```
